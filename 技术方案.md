# OSS / AWS S3 前端直传方案

前端统一 AWS SDK v3，**开发环境通过 OSS S3\-compatible endpoint 模拟 S3**。

## 一、背景

当前个人 AWS 账号申请成本较高，因此开发阶段用阿里云 OSS 的 S3 兼容接口来模拟 AWS S3；但从前端实现角度看，最终要尽量保持**一套上传代码**同时覆盖 OSS 和 AWS S3。

## 二、结论

### 主方案

前端统一使用** AWS SDK v3 \+ ****`@aws-sdk/lib-storage`** 作为上传实现。

- 开发环境：后端签发 OSS 临时凭证，并返回 OSS 的 S3\-compatible `endpoint`

- 生产环境：后端签发 AWS STS / AssumeRole 凭证，不返回 `endpoint`

- 前端代码保持不变，只切换后端返回的 token 内容

### 不作为主架构的方案

不建议把“开发用 `ali-oss`、生产用 `@aws-sdk/*`”作为主上传架构。

原因很简单：

- 两套 SDK 的 multipart、进度、URL、重试语义不完全一致

- 后端联调契约会被两套实现拉扯，后续更容易产生 drift

- OSS 官方文档已经**明确支持用 AWS SDK v3 访问 OSS**

## 三、落地建议

实际落地时，至少拆成下面几层：

```Plaintext
src/
├── upload/
│   ├── client.ts        # S3Client 初始化
│   ├── types.ts         # UploadTokenResponse 等类型
│   ├── upload.ts        # Upload / multipart 封装
│   └── policy.ts        # 可选：POST Policy 相关逻辑
├── hooks/
│   └── useUpload.ts     # 业务层 Hook
└── components/
    └── Uploader.tsx     # 上传组件
```

## 四、为什么选“统一 AWS SDK v3 \+ OSS 兼容 endpoint”

1. **开发和生产的语义更一致**

    同一套 `S3Client` / `Upload` 逻辑，开发时切 OSS endpoint，生产时切 AWS 默认 endpoint。

2. **更适合和公司后端对接**

    公司后端未来大概率会给出 AWS STS / S3 风格的临时凭证，前端无需重写上传逻辑。

3. **OSS 官方支持这条路**

    OSS 提供 S3 兼容接口，AWS SDK v3 可直接通过 `endpoint` \+ 临时凭证访问 OSS。

4. **避免双 SDK 漂移**

    双 SDK 方案会在 multipart、重试、URL 处理、resume 等行为上逐渐分叉，维护成本高。

## 五、统一上传契约

建议把新方案的后端返回值定义成“上传令牌”，而不是让前端感知具体云厂商细节。

```TypeScript
export interface UploadTokenResponse {
  provider: 'oss' | 'aws'
  accessKeyId: string
  secretAccessKey: string
  sessionToken: string
  expiration: string
  bucket: string
  region: string
  endpoint?: string
  keyPrefix?: string
  publicBaseUrl?: string
  objectUrl?: string
}
```

### 约束说明

- `region` 必须是统一后的 region 标识，不要把 `oss-cn-*` 和 `cn-*` 混着当同一个值用

- OSS 侧可以在后端内部继续保留 `OSS_REGION` 之类的 env 命名，但返回给前端前要做归一化

- `endpoint` 只表示浏览器上传的基础 host，OSS dev 时返回，AWS prod 时省略

- `keyPrefix` 用于约束前端可写入的对象路径，避免前端无限制写任意目录

- `publicBaseUrl` 只用于公开读对象的展示地址前缀，不要让前端自己猜最终 URL

- `objectUrl` 只在后端已经确定最终访问地址时返回；如果对象走签名下载、CDN 或私有桶，应优先由后端直接给出

## 六、上传流程

1. 前端选择文件

2. 请求后端 `GET /api/upload-token`

3. 后端返回临时凭证、region、bucket、可选 endpoint / keyPrefix / publicBaseUrl / objectUrl

4. 前端初始化 `S3Client`

5. 根据后端约定生成 `objectKey`（优先使用后端返回的 `keyPrefix` 约束路径）

6. 调用 `Upload` 执行上传（小文件简单上传，大文件 multipart）

7. 监听进度更新 UI

8. 上传完成后，前端把 `key` 提交给业务后端保存

### 前端示意

```TypeScript
import { S3Client } from '@aws-sdk/client-s3'
import { Upload } from '@aws-sdk/lib-storage'

const token = await fetch('/api/upload-token').then(r => r.json())

const client = new S3Client({
  region: token.region,
  ...(token.endpoint ? { endpoint: token.endpoint } : {}),
  credentials: {
    accessKeyId: token.accessKeyId,
    secretAccessKey: token.secretAccessKey,
    sessionToken: token.sessionToken,
  },
})

const objectKey = `${token.keyPrefix ?? 'uploads/'}${file.name}`

const upload = new Upload({
  client,
  params: {
    Bucket: token.bucket,
    Key: objectKey,
    Body: file,
    ContentType: file.type,
  },
  // 分片大小/阈值：Upload 按该值切块；<= 10 MiB（约 10 MB）只有一个块，走 PutObject 简单上传
  // > 10 MiB 会产生多个块，自动走 multipart upload
  partSize: 10 * 1024 * 1024,
  // 分片并发数：仅 multipart 时生效；失败重试依赖 S3Client retry 策略，刷新后续传仍需单独 checkpoint
  queueSize: 4,
})
```

## 七、后端联调契约

至少要跟后端提前约定下面几件事：

### 凭证格式

- 返回稳定 DTO

- 字段名固定，建议统一使用 `accessKeyId` / `secretAccessKey` / `sessionToken` / `expiration`

- `endpoint`、`region` 必须由后端显式返回与当前 Bucket 完全匹配的值，前端不自行推导

- 不在前端暴露长期 AccessKey / SecretKey

### 对象 key 归属

二选一即可，不要混用：

- 后端直接返回完整 `objectKey`

- 后端返回 `keyPrefix`，前端只在这个前缀下拼接文件名

### 可访问性

- 如果对象是公开读，后端应返回 `publicBaseUrl`

- 如果后端已经能确定最终访问地址，可以直接返回 `objectUrl`

- 如果对象是私有读，前端不要自己拼“可访问 URL”，应走后端签名下载 / 预览接口

### CORS

Bucket 必须允许浏览器直传所需的 origin 和方法，尤其是 multipart 上传所需的 `PUT` / `POST` / `HEAD` 等。

### TTL / 刷新

- 上传 token 必须有明确过期时间

- token TTL 要覆盖预期上传时长，或后端提供刷新机制

- 前端不要假设页面刷新后还能自动续上同一次上传

### OSS 兼容 S3 的边界与注意事项

OSS 可以看作 **S3 兼容子集**，前端直传可用，但不是和 Amazon S3 完全等价。需要特别注意：

|差异点|说明|
|---|---|
|请求风格|OSS 只支持 `Virtual Hosted`，Bucket 必须放在子域名里；不要走 `forcePathStyle: true` 这类 path\-style。|
|Endpoint / Region / 域名|OSS 与 AWS 的 `region`、`endpoint`、最终公网访问域名不是同一件事；开发联调用 OSS 时要显式下发匹配的 `endpoint`，新开通中国内地 OSS 还可能需要自定义域名 / CNAME。|
|认证体系|前端不能混用 AWS 和 OSS 的长期密钥；浏览器场景应由后端签发 OSS STS 临时凭证，前端把 `securityToken` 映射到 AWS SDK v3 的 `sessionToken`。|
|ACL 语义|OSS 只支持 `private`、`public-read`、`public-read-write` 三种 ACL 模式，语义也不完全等同于 S3。|
|存储类型|OSS 的 `Standard / IA / Archive` 对应 S3 的不同存储类，但 `Archive` 恢复天数与 S3 不完全一致。|
|ETag|OSS 与 S3 的 ETag 大小写和分片计算方式都不同，不能拿它当跨云一致的完整性或去重标准。|
|`x-oss-process`|通过 S3 协议访问 OSS 时，`x-oss-process` 仅支持 `image/` 和 `style/`，不支持 `video/` 等处理类型。这里限制的是 OSS 的对象处理能力，不是视频文件上传能力；视频文件本身照样可以通过 `PutObject` 或 multipart 正常上传，只是不能指望在这条兼容链路上顺带完成转码、截帧、水印等处理。|
|断点续传|两边都基于 multipart 支持恢复上传。差别主要在 SDK 封装：`ali-oss` 更直接暴露 checkpoint，而浏览器侧 `@aws-sdk/lib-storage` 默认不持久化上传会话。|

> 结论：如果只做前端直传上传，`PutObject` / multipart 这条主链路是可用的；如果要依赖视频处理、Archive 恢复、ACL 细粒度语义或刷新后自动续传，就不能把 OSS 和 S3 视为完全等价。

### 风险与注意事项

|约束|说明|
|---|---|
|OSS endpoint 格式|应使用 `https://s3.oss-{region}.aliyuncs.com`，不是 `https://s3.{region}.aliyuncs.com`|
|Region 语义|OSS / AWS 的 region 不要直接混用，token DTO 要做归一化|
|公网 URL|不要在前端靠 `bucket + endpoint` 盲拼最终访问地址，尤其是 OSS 自定义域名 / CNAME 场景|
|ETag|不要把 multipart ETag 格式当作跨云一致的成功标志|
|断点续传|`@aws-sdk/lib-storage` 默认不持久化 checkpoint / 上传会话；如果业务强依赖“页面刷新后继续上传”，就不能只依赖默认封装，要么自己设计 multipart 会话持久化与恢复逻辑，要么继续评估具备现成 checkpoint 能力的 OSS SDK / 实现。|
|OSS 新域名规则|2025\-03 之后中国内地新 OSS 用户可能需要自定义域名访问数据 API|
